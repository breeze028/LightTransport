# CWBVH：从论文到 CUDA Wavefront 实现

CWBVH（Compressed Wide BVH）是 CUDA wavefront 路径追踪器内部使用的压缩 8-wide BVH traversal layout。它的目标不是改变路径积分器，也不是新增用户可选的加速结构模式，而是在不改变命中语义的前提下降低 GPU 软件遍历的内存流量，尤其针对 secondary rays、shadow rays、ReSTIR visibility 这类方向不再相干的射线。

本文按 Ylitie、Karras、Laine 的 *Efficient Incoherent Ray Traversal on GPUs Through Compressed Wide BVHs* 思路讲到当前 LightTransport 实现。理论部分讲 CWBVH 为什么能降低带宽、80-byte node 如何编码、压缩 stack 如何工作；工程部分以当前源码为准，说明 tinybvh 转换、GPU 解码遍历、BVH8/binary fallback、Wavefront/ReSTIR 影响面和验证门槛。

## 1. 范围与资料

当前实现只作用于 CUDA wavefront 的软件 traversal：

- 不改变 CPU path tracer。
- 不改变 CUDA megakernel 的用户可见行为。
- 不新增 `AccelerationStructure` 枚举值。
- 不新增 `--accel cwbvh`、编辑器 checkbox 或调试开关。
- 只在 CUDA wavefront + Two-level BLAS 场景下作为内部 layout 候选。
- 如果 CWBVH 构建失败或某个 mesh 没有 `cwbvh_root`，会 fallback 到 BVH8，再 fallback 到 compact binary BLAS。

主要源码：

- `src/gpu/scene_upload.cuh`
- `src/gpu/intersection.cuh`
- `src/gpu/types.cuh`
- `src/gpu/cuda_path_tracer.cu`
- `include/lt/renderer.h`
- `external/tinybvh/tiny_bvh.h`

参考资料：

- Ylitie, Karras, Laine, *Efficient Incoherent Ray Traversal on GPUs Through Compressed Wide BVHs*, HPG 2017。
- tinybvh `BVH8_CWBVH`：80-byte 8-wide BVH 表示，基于 Ylitie et al. 和 AlanWBFT 的实现。
- AlanWBFT/CWBVH：Ylitie 论文的开源实现之一。

## 2. Section 1：问题是什么

普通二叉 BVH traversal 对每条 ray 反复执行：

```text
load node
test left AABB
test right AABB
push/pop stack
eventually test triangles
```

对 primary camera rays，这通常还不错，因为相邻线程射线方向接近、访问节点相近、cache 命中较好。路径追踪越往后走，ray 越不相干：

- diffuse bounce 的方向分散。
- glossy/rough reflection 只部分相干。
- shadow ray 指向不同光源或不同 reservoir sample。
- ReSTIR GI/PT 的 secondary sample 和 replay path 会访问分散区域。

这时 traversal 的瓶颈经常不是三角形相交本身，而是“每条 ray 随机读取很多 BVH node”。Ylitie 等人的核心判断是：对 incoherent rays，减少 hierarchy memory traffic 比保持 full-float 节点更重要。

CWBVH 做两件事：

1. 把树变宽：一次读取一个 8-wide 节点，测试最多 8 个 child，减少树深度和 stack 操作。
2. 把节点压缩：child bounds 用父节点局部坐标和 8-bit 量化表示，让一个 8-wide node 约 80 字节。

论文报告的收益来自这两个方向叠加：更少节点访问、更小节点、更紧凑 stack。代价是每个节点需要额外 bit decode、量化 bounds 还原和 hit mask 操作。它不是“永远更快”的数据结构；它是一个用计算换带宽的 traversal layout。

## 3. Section 2：从二叉 BVH 到 Wide BVH

二叉 BVH 每个 interior node 有 2 个 child。Wide BVH 把多个层级折叠成一个节点：

```text
binary subtree
  -> collect up to 8 child subtrees/leaves
  -> one BVH8 node
```

一次 node test 会得到一个 child hit mask：

```text
slot 0 hit? slot 1 hit? ... slot 7 hit?
```

然后 traversal 根据 ray direction 和 child ordering 选择访问顺序。Wide BVH 的好处：

- 树深度更浅。
- stack push/pop 次数更少。
- 一次 node load 能覆盖更多候选 child。
- GPU warp 中即使每条 ray 走不同 child，也可能少读几层中间节点。

坏处：

- 每个节点本身更大。
- 如果 full-float 存 8 个 child AABB，带宽很快变贵。
- 多 child 排序和 stack 表示更复杂。

因此 CWBVH 不是单纯“BVH8”，而是“压缩后的 BVH8”。

## 4. Section 3：80-byte 压缩节点

当前项目使用 tinybvh `BVH8_CWBVH` 布局，并在 GPU 端保存为：

```cpp
struct GpuCwBvhNode {
    float4 block[5];
};
```

`sizeof(GpuCwBvhNode) == 80`。五个 `float4` 不是语义上的 float 数组，而是 16-byte block。部分字段把整数 bit pattern 塞进 float lane，host 端用 `cwbvh_uint_to_float()` 写入，device 端用 `__float_as_uint()` 读回。

节点包含：

```text
block[0].xyz  : nodeLo，量化坐标原点
block[0].w    : ex, ey, ez, imask
block[1].x    : childBase
block[1].y    : triangleBase
block[1].z/w  : 8 个 child meta byte
block[2..4]   : qlo/qhi bytes for x/y/z, 8 slots
```

`imask` 标记哪些 slot 是 internal child。meta byte 解释为：

```text
meta == 0
  -> empty slot

slot in imask
  -> internal child
  -> meta 高位用于类型标记，低位辅助生成 node hit bit

slot not in imask
  -> leaf triangle group
  -> 高 3 bit 是 unary triangle count
  -> 低 5 bit 是 triangle group 起始偏移
```

这个编码让 traversal 不需要为每个 child 保存完整 pointer。internal children 被连续存储，`childBase + popcount(mask before slot)` 就能得到 child node index。leaf triangles 也按节点局部顺序连续存储，用 `triangleBase + triBit * 3` 找到三角形 payload。

## 5. Section 4：Bounds 量化

每个 child AABB 不再保存 full-float min/max，而是在父节点局部空间中量化到 8-bit：

```text
qlo = floor((child_min - nodeLo) / scale)
qhi = ceil ((child_max - nodeLo) / scale)
scale = 2^e
```

每个轴有独立 exponent：

```text
sx = 2^ex
sy = 2^ey
sz = 2^ez
```

`floor`/`ceil` 很关键：压缩后的 bounds 必须保守包住原始 bounds。量化可以让盒子略微变大，导致多做一些 child/triangle test；不能让盒子变小，否则会漏交。

GPU 遍历时不一定显式还原 full-float bounds。当前 `intersect_cwbvh_quantized_child_gpu()` 把 ray-box slab test 改写为：

```text
orig_axis = (nodeLo_axis - ray_origin_axis) * inv_dir_axis
idir_axis = scale_axis * inv_dir_axis
t0 = qmin * idir_axis + orig_axis
t1 = qmax * idir_axis + orig_axis
```

这样 child bounds 以 byte 形式读取后直接进入 t 区间计算。根据 ray direction 正负，`qlo/qhi` 会交换成近端/远端：

```text
qmin = direction_negative ? qhi : qlo
qmax = direction_negative ? qlo : qhi
```

最终：

```text
tmin = max(tx0, ty0, tz0, ray_tmin)
tmax = min(tx1, ty1, tz1, current_closest_t)
hit  = tmax >= tmin
```

## 6. Section 5：Node Group、Triangle Group 与压缩 Stack

Ylitie 论文里一个重要技巧是 stack entry 不保存普通 node pointer，而保存“待访问的一组 node”或“待访问的一组三角形”。当前代码沿用 tinybvh 风格，用两个 `uint2`：

```text
ngroup = { base, mask }
tgroup = { base, mask }
```

其中：

- `ngroup` 表示一组 internal children。
- `tgroup` 表示一组 leaf triangle hits。
- `base` 是连续存储区域的起点。
- `mask` 标记哪些 slot/triangle 需要访问。

当前设备端用高 8 bit 区分 node group：

```text
ngroup.y > 0x00ffffffu  -> node group
ngroup.y <= 0x00ffffffu -> triangle group 或 empty
```

访问 internal child 时：

```text
child_bit_index = highest_set_bit(ngroup.y)
slot_index = (child_bit_index - 24) ^ octinv
relative_index = popcount(imask & lower_slot_mask)
node_index = childBase + relative_index
```

`octinv` 由 ray direction sign 得到，用来调整 slot 顺序，使 traversal 更倾向先访问 ray 方向上的近端 child。它不是严格按 t 排序，但成本低、与 slot 编码匹配。

如果一个 node group 还有未访问 child，就把剩余 group push 到短 stack。当前 GPU 实现使用：

```cpp
uint2 stack[32];
```

这比为每个 child 单独 push pointer 更紧凑。代价是 traversal 顺序和 metadata 编码必须完全一致；child base、imask、meta 任意一个错，表现通常是漏交、黑块或随机错误。

## 7. Section 6：Triangle Payload

CWBVH 的 leaf triangles 独立存储在：

```text
GpuScene::cwbvh_triangles
```

当前每个三角形占 3 个 `float4`：

```text
edge2_block = cwbvh_triangles[tri_addr + 0]
edge1_block = cwbvh_triangles[tri_addr + 1]
v0_block    = cwbvh_triangles[tri_addr + 2]
```

语义是：

```text
edge2_block.xyz -> edge2
edge2_block.w   -> material_and_flags bits
edge1_block.xyz -> edge1
v0_block.xyz    -> v0
v0_block.w      -> global triangle index bits
```

设备端临时组出 `GpuTraversalTriangle`：

```cpp
const GpuTraversalTriangle traversal_tri{
    Vec3{v0_block.x, v0_block.y, v0_block.z},
    Vec3{edge1_block.x, edge1_block.y, edge1_block.z},
    Vec3{edge2_block.x, edge2_block.y, edge2_block.z},
    material_and_flags,
};
```

然后走现有 Moller-Trumbore triangle test。命中后写 `GpuCompactHit`：

```text
t, u, v
material_and_flags
global triangle index
sphere = -1
```

这保持了当前 wavefront 的“traversal hit”和“shading hit”分离：intersect kernel 不读取完整 `GpuTriangle` 的 normal、UV、tangent、emission、lightmap UV。只有 closest compact hit 确认后，后续 shading 才恢复完整数据。

## 8. Section 7：论文方案和本项目的差异

Ylitie 等人的论文重点是高质量 CWBVH 构建、压缩节点格式、压缩 traversal stack，以及 incoherent ray 上的 GPU traversal 性能。tinybvh 的 `BVH8_CWBVH` 是一个实用参考实现：先构建 BVH，再折叠到 `MBVH<8>`，最后转换成 CWBVH 数据。

LightTransport 当前没有完整复刻论文的构建优化路线，而是采用工程上更保守的接入方式：

1. `RenderScene` 仍先构建项目自己的二叉 BLAS/TLAS。
2. CUDA upload 阶段对每个 mesh BLAS 子树收集三角形。
3. 调用 tinybvh `BVH8_CWBVH::Build()` 生成 CWBVH。
4. 复制 tinybvh `bvh8Data` 到 `GpuCwBvhNode`。
5. 复制 tinybvh `bvh8Tris` 到 `GpuScene::cwbvh_triangles`。
6. 修正 child base 和 triangle base，因为多个 mesh 的 CWBVH buffer 被合并到全局数组。
7. 把 tinybvh 的本地 triangle index remap 回 LightTransport 的全局 triangle index。
8. 给 triangle payload 写入 LightTransport 的 `material_and_flags`，保留 alpha fast path 语义。

这种做法的优点是接入快、正确性容易和 tinybvh 对齐。缺点是构建阶段多了一套 BVH 构建/转换，且 CWBVH 树质量、triangle ordering、node ordering 不一定最适合当前 renderer 的 ray distribution。

## 9. 当前打包流程

`pack_scene_from_render_scene()` 中和 CWBVH 相关的逻辑：

```text
gpu.use_two_level = use_two_level_accel(...)
pack_wide_bvh = use_wavefront_bvh8_layout(...)
pack_cwbvh = use_wavefront_cwbvh_layout(...)

build compact binary traversal nodes
if pack_wide_bvh:
  build_traversal_bvh8_node(...)
if pack_cwbvh:
  build_tinybvh_cwbvh_node(...)

mesh_instance = {
  bvh_root,
  bvh8_root,
  cwbvh_root,
}
```

CWBVH 只在这些条件满足时尝试打包：

- `settings.cuda_wavefront == true`
- `AccelerationStructure::TwoLevel`
- CWBVH 节点和 triangle payload 的估算内存不超过 64 MiB
- 当前内部实验开关允许

`build_tinybvh_cwbvh_node()` 的关键步骤：

```text
collect subtree triangle indices from binary BLAS root
build tinybvh vertex array
local_to_global[local] = global triangle index
tinybvh::BVH8_CWBVH::Build(vertices, tri_count)
copy bvh8Data as 5 float4 blocks per node
patch childBase by global node offset
copy bvh8Tris as 3 float4 blocks per triangle
patch v0.w from local triangle index to global triangle index
patch triangleBase by global triangle payload offset
later: write material_and_flags into edge2.w
```

这里有三个容易出错的偏移：

- `childBase`：tinybvh 单个 BLAS 内的 node offset，要加全局 `node_offset`。
- `triangleBase`：tinybvh 单个 BLAS 内的 triangle payload offset，要加全局 `tri_float4_offset`。
- `v0.w`：tinybvh local triangle index，要通过 `local_to_global` 改回全局 triangle index。

## 10. 当前设备端 traversal

CUDA kernel 通过模板选择 traversal layout：

```text
GpuTraversalLayout::Binary
GpuTraversalLayout::Bvh8
GpuTraversalLayout::CwBvh
```

Two-level + CWBVH 的大致流程：

```text
intersect compact ray
  -> traverse binary TLAS
     -> leaf instance
        -> if instance.cwbvh_root >= 0:
             intersect CWBVH BLAS
           else if instance.bvh8_root >= 0:
             intersect BVH8 BLAS
           else:
             intersect binary BLAS
  -> intersect analytic spheres
```

CWBVH 设备端入口：

```text
intersect_blas_cwbvh_compact_gpu()
  -> intersect_cwbvh_compact_gpu()
       -> decode node
       -> quantized child tests
       -> produce hitmask
       -> visit node groups / triangle groups
       -> intersect_cwbvh_triangle_compact_gpu()
```

当前 CWBVH 只用于 compact hit path。普通 full `GpuHit` 路径仍可走二叉结构；这是为了保持 shading 数据恢复路径集中在 wavefront stage 里。

## 11. 用户可见语义

公开加速结构仍只有：

```text
Flat
TwoLevel
```

CLI 接受：

```text
--accel flat
--accel two-level
--accel twolevel
--accel tlas
```

旧 `Auto` 加速结构模式已经移除。`--accel auto` 会被当作未知值报告，并保持默认 `TwoLevel` 不变。

编辑器中：

- CPU 和 CUDA Megakernel 可选择 `Flat BVH` 或 `Two-level BVH`。
- CUDA Wavefront 固定显示 `Two-level BVH (Wavefront internal)`。
- BVH8/CWBVH 不出现在 UI 中。

原因是 BVH8/CWBVH 是 renderer 内部实现细节。公开它会让用户以为它们是稳定功能，但当前它们仍需要按场景和 profile 数据决定是否值得启用。

## 12. 与 Wavefront/ReSTIR 的关系

CWBVH 不是单独 renderer。它服务于 CUDA wavefront 的所有 trace 类 kernel：

- 普通 wavefront intersect。
- shadow visibility。
- SVGF first-hit AOV 所依赖的 primary compact hit。
- ReSTIR DI visibility。
- ReSTIR GI secondary trace。
- ReSTIR PT replay、visibility 和 continuation trace。

这有两个后果：

1. CWBVH 的正确性错误会被 ReSTIR history 放大。一次 visibility 漏交可能稳定进入 reservoir/history，表现比普通路径追踪更“持久”。
2. CWBVH 的性能收益必须看全帧。单个 secondary trace kernel 变快，但如果 upload、register pressure、occupancy 或 fallback 让其他 stage 变慢，最终仍不该默认启用。

## 13. Profiling 门槛

CWBVH 的预期收益来自“少读内存”，但当前项目里它还要支付：

- tinybvh 构建/转换成本。
- 80-byte node decode 成本。
- meta/imask/hitmask bit 操作。
- `__clz`/`__popc`/stack group 管理。
- triangle payload remap。
- 可能更差的 traversal ordering。

因此验收门槛应以稳定帧为准：

```text
Sponza median frame time improves >= 5%
P95 frame time does not regress
image correctness matches BVH8/binary fallback
```

建议 workload：

```powershell
.\build\Release\lt_render.exe .\scenes\Sponza\Sponza.lt .\build\cwbvh_sponza.png --cuda --cuda-wavefront --size 1010 789 --spp 1 --frames 64 --max-bounces 2 --denoiser svgf --aa taa
```

分析时至少看：

- `wavefront_intersect_kernel`。
- shadow/visibility trace kernel。
- ReSTIR GI/PT secondary trace kernel。
- H2D upload 是否只在首帧。
- CWBVH buffer 是否每帧重建。
- fallback 比例是否异常高。
- register count 降低是否真的转化成 frame time 降低。

Nsight Compute 若显示 register count 下降但 frame time 上升，说明压缩节点没有自动赢。继续看 memory stall、instruction count、occupancy、L1/L2 hit rate 和 branch behavior。

- 没有黑帧。
- 没有随机漏面。
- alpha 材质可见性与 BVH8/binary 对照一致。
- SVGF normal/depth/object-id debug view 没有大块破损。
- ReSTIR DI/GI/PT 没有明显 visibility 漏光。

1. `GpuCwBvhNode` 仍为 80 字节。
2. host 端 tinybvh block copy 与 device 端 byte offset 一致。
3. `childBase` 加了全局 node offset。
4. `triangleBase` 加了全局 triangle payload offset。
5. `v0.w` 已从 local triangle index 改为 global triangle index。
6. `edge2.w` 写入了 `material_and_flags`。
7. `GpuMeshInstance::cwbvh_root` 对每个 instance 正确。
8. CWBVH 构建失败时 BVH8/binary fallback 保留。
9. 所有 trace launch 的 `GpuTraversalLayout::CwBvh` 分支一致。

## 15. 常见问题

- `--accel cwbvh` 不生效：预期行为。CWBVH 不是公开 acceleration mode。
- Wavefront 选中后不能改 `Acceleration`：预期行为。编辑器固定 Two-level，并把 wide layout 交给 CUDA backend 内部决定。
- CWBVH buffer 已上传但没有性能收益：检查 kernel 是否实际选择 `GpuTraversalLayout::CwBvh`，以及 instance 是否因为 `cwbvh_root < 0` fallback。
- 图像出现漏光或黑块：优先查 triangle index remap、triangle base、child base、meta leaf count、alpha flags。
- Sponza 变慢但 register count 降低：说明 decode/bit 操作/访问顺序成本超过带宽收益，不能只凭 register count 合入默认路径。
- Cornell 正确但 Sponza 错：大场景更容易暴露 offset 溢出、multi-mesh root map、triangle payload 合并和 fallback 路径问题。

## 16. 当前结论

CWBVH 在 LightTransport 中应保持“内部实验 traversal layout”的定位。它的理论动机非常明确：对 incoherent GPU rays，用压缩 wide node 减少 hierarchy memory traffic。但当前 renderer 还有自己的约束：

- TLAS 仍是二叉。
- BLAS 来自现有 `RenderScene` 再经 tinybvh 转换。
- Wavefront/ReSTIR 多个 kernel 共用 traversal。
- correctness fallback 比暴露用户开关更重要。
- 默认启用必须由 Sponza 等稳定 workload 的 median/P95 数据证明。

如果后续要继续推进，优先方向不是增加 UI 开关，而是把 profile 闭环做扎实：确认 CWBVH 真实命中、减少 fallback、比较 BVH8 和 CWBVH 的 full-frame cost，并用 ReSTIR/SVGF 场景验证 visibility 和 history 不被破坏。
